## 1. 용어 정리

### 엔티티(Entity)란?

그냥 **"데이터 덩어리 객체"**. 예를 들면:

```python
class User:
    id: int
    name: str
    email: str
```

### 액티브 레코드 패턴

**"DB 테이블 한 행 = 객체 하나"** 인데, 그 객체가 **저장/수정/삭제까지 스스로 함**.

```python
class User:
    id: int
    name: str
    
    def save(self):      # 자기가 직접 DB에 저장
        db.execute("INSERT INTO users ...")
    
    def delete(self):   # 자기가 직접 DB에서 삭제
        db.execute("DELETE FROM users ...")
```

객체가 **데이터 + DB 조작** 둘 다 들고 있는 것.

### 도메인 모델

**"비즈니스 규칙을 코드로 표현한 것"**

예를 들어 "회원 탈퇴하면 주문도 취소돼야 한다" 같은 규칙이 도메인 로직임. 비즈니스 로직이 여러 객체에 분산되어 있고 객체들끼리 서로 협력하는 구조.

## 2. 계층형 아키텍처

### 기본 구조

```
[표현 계층] ← 화면/API
     ↓
[비즈니스 계층] ← 규칙/로직
     ↓
[데이터 계층] ← DB 저장/조회
```

### 언제 잘 맞나?

| 패턴 | 로직 복잡도 | 계층형 아키텍처 적합성 |
|---|---|---|
| 트랜잭션 스크립트 | 단순 | ✅ 잘 맞음 |
| 액티브 레코드 | 중간 | ✅ 잘 맞음 |
| 도메인 모델 | 복잡 | ❌ 어울리지 않음 |

### 왜 도메인 모델에는 안 맞냐?

도메인 모델은 비즈니스 로직이 **여러 객체에 분산**되어 있어.

```
Order 객체 → OrderItem 객체 → Product 객체 → Discount 객체 ...
```

"주문 총액 계산" 하나를 위해 여러 객체가 서로 메서드를 호출하고 협력함. 이러면 **어느 계층에 뭘 넣어야 하는지** 경계가 모호해져서 계층 경계를 넘나들게 됨.

반면 **트랜잭션 스크립트**는 한 함수가 처음부터 끝까지 절차적으로 흘러가서 계층 구분이 쉬움:

```python
def process_order(user_id, items):
    user = db.get(user_id)      # 조회
    total = sum(items)           # 계산
    db.save(order)               # 저장
    email.send(user)             # 알림
```

## 3. 인프라스트럭처 계층 통합

### 왜 통합했나?

원래 계층형 아키텍처:
```
프레젠테이션 계층 (UI)
비즈니스 로직 계층
데이터 접근 계층 (DB)
```

생각해보면 **프레젠테이션(UI)** 이랑 **데이터 접근(DB)** 이 하는 일이 비슷해.

- UI → 외부 사용자와 연결
- DB → 외부 데이터베이스와 연결
- 외부 서비스 → 외부 API와 연결

**둘 다 "외부랑 연결하는 것"**. 비즈니스 로직이 아니라 기술적인 연결 담당.

그래서 "어차피 같은 역할이니까 **인프라스트럭처 계층** 하나로 묶자" 한 것.

```
인프라스트럭처 계층: [데이터베이스 | UI 프레임워크 | 외부 제공자 | 메시지 버스]
```

## 4. DIP (의존성 역전 원칙)

### 기존 문제 (뒤집기 전)

```
비즈니스 로직 계층
        ↓ 의존
인프라스트럭처 계층 (DB, UI...)
```

비즈니스 로직이 인프라에 의존하면:

```python
class OrderService:  # 비즈니스 로직
    def place_order(self):
        db = MySQLDatabase()  # DB를 직접 알고 있음!
        db.save(order)
```

**MySQL을 PostgreSQL로 바꾸면? → OrderService 코드도 바꿔야 함.**
비즈니스 로직이 DB 기술에 종속된 것.

### DIP 적용 후 (뒤집으면)

```
비즈니스 로직 계층
        ↑ 의존 (화살표 반대!)
인프라스트럭처 계층
```

```python
class OrderService:  # 비즈니스 로직
    def place_order(self, repository):  # 그냥 "저장소"만 알면 됨
        repository.save(order)

# 실제 DB는 인프라가 알아서 처리
class MySQLRepository:
    def save(self, order): ...

class PostgreSQLRepository:
    def save(self, order): ...
```

이제 **DB를 바꿔도 비즈니스 로직은 안 건드려도 됨!**
인프라가 비즈니스 로직에 맞춰주는 구조.

## 5. 애플리케이션 계층

### 그림 8-10 구조

```
비즈니스 로직 계층  ← 핵심, 아무것도 안 의존
        ↑
애플리케이션 계층  ← "어떤 순서로 실행할지" 조율
        ↑
인프라스트럭처 계층  ← DB, UI, 외부API 다 여기
```

### UI가 밑에 있는 이유

"위에 있다 = 더 중요하다, 핵심이다"

UI는 그냥 **입출력 도구**야. 웹으로 바꾸든, 앱으로 바꾸든 비즈니스 로직은 안 바뀌잖아. 그러니까 UI가 밑(인프라)에 있는 게 맞아.

### 애플리케이션 계층이 하는 일

```python
class PlaceOrderAction:  # 애플리케이션 계층
    def execute(self, user_id, items):
        # 1. 유저 조회해서
        # 2. 주문 생성하고  
        # 3. 결제 처리하고
        # 4. 알림 보내고
        # → 순서 조율만 함, 비즈니스 규칙은 모름
```

비즈니스 규칙(할인 계산 등)은 모르고, **"어떤 순서로 뭘 호출할지"만** 담당.

### 전체 목적 요약

| 문제 | 해결책 |
|---|---|
| UI/DB가 계층마다 따로 있어서 복잡 | 인프라 계층으로 통합 |
| 비즈니스가 DB 기술에 종속 | DIP로 의존 방향 뒤집기 |
| 비즈니스 로직이 흐름 조율까지 담당 | 애플리케이션 계층이 조율 분리 |

결국 다 **"비즈니스 로직을 건드리지 않고 나머지를 갈아끼울 수 있게"** 하려는 목적.

## 6. 포트와 어댑터 패턴

### 현실 비유

**콘센트(포트)와 플러그(어댑터)**:

- 벽에 있는 **콘센트 구멍** = **포트** (규격만 정의, "220V 두 구멍"이라는 약속)
- 나라마다 다른 **플러그 변환기** = **어댑터** (실제로 꽂히는 구현체)

핵심은 **콘센트(포트)는 절대 안 바뀌고, 플러그(어댑터)만 갈아끼워.**

### 코드로 보면

**포트 = 인터페이스 (약속)**

```python
# 비즈니스 로직이 정의하는 "약속"
class UserRepository:  # 포트
    def save(self, user): ...
    def find_by_id(self, id): ...
```

비즈니스 로직은 이 약속만 알아. 실제로 어떻게 저장하는지 몰라도 됨.

**어댑터 = 실제 구현체**

```python
# 인프라가 그 약속을 구현
class MySQLUserRepository(UserRepository):  # 어댑터
    def save(self, user):
        mysql.execute("INSERT INTO users ...")

class MongoUserRepository(UserRepository):  # 다른 어댑터
    def save(self, user):
        mongo.insert_one(user)
```

### 왜 필요한가?

**포트/어댑터 없을 때:**

```python
class OrderService:
    def place_order(self):
        mysql.execute(...)  # MySQL에 종속!
        gmail.send(...)     # Gmail에 종속!
        kakao.notify(...)   # 카카오에 종속!
```

DB 바꾸거나 알림 수단 바꾸면 **비즈니스 로직 코드를 직접 수정해야 함.**

**포트/어댑터 있을 때:**

```python
class OrderService:
    def place_order(self, repo, notifier):
        repo.save(order)      # 저장소가 뭔지 몰라도 됨
        notifier.send(user)   # 알림 수단이 뭔지 몰라도 됨
```

DB 바꿔도, 알림을 카카오→슬랙으로 바꿔도 **OrderService는 안 건드려도 됨.**

### 그림으로 보면

```
외부DB ↔ 어댑터 ↔ [포트 | 비즈니스 로직 | 포트] ↔ 어댑터 ↔ 외부API
```

- 비즈니스 로직은 **포트(구멍)만 뚫어놓고** 가운데 앉아있어
- 어댑터가 외부 세계와 비즈니스 로직 사이를 **번역**해줌
- 비즈니스 로직은 외부가 뭔지 전혀 몰라도 됨

### 언제 이 구조를 쓰나?

| 상황 | 이유 |
|---|---|
| DB를 나중에 바꿀 가능성 있을 때 | 어댑터만 교체 |
| 테스트 코드 짤 때 | 가짜 어댑터(Mock) 끼워서 테스트 |
| 알림 수단이 여러 개일 때 | 카카오/슬랙/이메일 어댑터 각각 구현 |
| 외부 API가 자주 바뀔 때 | 어댑터만 수정, 비즈니스 로직 무관 |

### 한 줄 요약

> **포트** = 비즈니스가 외부에 요구하는 "약속 문서"
> **어댑터** = 그 약속을 실제로 이행하는 "구현체"
> 목적 = 비즈니스 로직을 외부 기술 변화로부터 보호

## 7. 폴리글랏 모델링

**하나의 시스템에서 목적에 맞게 DB를 여러 개 쓰는 것.**

```
하나의 시스템에서:
- 실시간 데이터 처리 → MongoDB (문서 저장소)
- 분석/보고서       → Redshift (컬럼 저장소)
- 검색 기능         → Elasticsearch (검색 엔진)
```

포트-어댑터 구조에서는 **각 DB마다 어댑터를 따로 만들면** 됨. 비즈니스 로직은 그냥 포트(약속)만 호출.

### 왜 필요하냐?

완벽한 데이터베이스는 없어. 확장성, 일관성, 지원하는 질의 모델 간에 균형이 필요하거든. 그래서 각 요구사항에 가장 잘 맞는 DB를 골라서 쓰는 거야.

## 8. CQRS 패턴

### CQRS = Command Query Responsibility Segregation

**"쓰기(Command)"와 "읽기(Query)"를 분리하는 패턴**

### 왜 분리하냐?

```
일반적인 경우:
주문 저장할 때 → 주문 테이블에 씀
주문 조회할 때 → 같은 주문 테이블에서 읽음
```

근데 현실에서 **쓰기용 데이터 구조**랑 **읽기용 데이터 구조**가 달라.

```
쓰기: 정규화된 테이블 (데이터 정합성 중요)
읽기: 조인 다 풀어놓은 뷰 (속도 중요)
```

그래서 아예 분리해버리는 거야:

```
사용자가 주문 → Command → 쓰기 DB에 저장
사용자가 조회 → Query   → 읽기 DB에서 빠르게 반환
```

### CQRS가 폴리글랏(여러 DB) 동기화 문제를 해결하는 이유

폴리글랏으로 DB 여러 개 쓰면 동기화 문제가 생겨:
```
사용자가 주문함
    ↓
MySQL에 저장 ✅
MongoDB에도 저장해야 함 🤔
Elasticsearch에도 저장해야 함 🤔
```

CQRS는 이렇게 해결해:

```
주문 생성 (Command)
    ↓
쓰기 DB에만 저장 (MySQL)
    ↓
"주문생성됨" 이벤트 발행
    ↓
읽기 DB들이 이벤트 받아서 각자 알아서 업데이트
    ├── MongoDB 업데이트
    ├── Elasticsearch 업데이트
    └── Redis 업데이트
```

비즈니스 로직이 각 DB를 직접 챙기지 않아도 됨. 이벤트 하나 던지면 각 DB가 알아서 받아서 업데이트.

**비유:** CQRS 없을 때는 사장이 직접 창고/매장/온라인몰에 재고 하나씩 전화해서 업데이트. CQRS 있을 때는 사장이 "재고 변경됨" 공지 하나 올리면 창고/매장/온라인몰이 각자 알아서 반영.

## 9. 이벤트 소싱

### 일반적인 저장 방식 vs 이벤트 소싱

**일반 저장 방식:**
```
주문 상태: "배송중"  ← 현재 상태만 저장
```

**이벤트 소싱 방식:**
```
1. "주문 생성됨" (10:00)
2. "결제 완료됨" (10:05)
3. "배송 시작됨" (14:00)
← 모든 변화 이력을 이벤트로 저장
```

**현재 상태 = 이벤트들을 순서대로 재생한 결과**

은행 통장이랑 똑같아:
- 잔액 1만원 → 일반 저장
- 입금 3만원, 출금 2만원, 입금 1만원 → 이벤트 소싱

### 이벤트 소싱의 약점

조회가 불가능해. 이유는:

```
이벤트 소싱 DB:
| user_id | 이벤트          | 시간  |
|---------|----------------|-------|
|    1    | 회원가입        | 10:00 |
|    1    | 포인트 +1000   | 10:05 |
|    1    | 포인트 +500    | 11:00 |
|    1    | 포인트 -200    | 12:00 |
```

현재 포인트가 얼마인지 테이블 어디에도 없어. 1000 + 500 - 200 = 1300이라는 걸 알려면 이벤트를 **전부 다 재생**해야 해. 그러니까 "포인트 1000 이상인 유저 찾아줘" 같은 쿼리가 **불가능**.

## 10. 애그리게이트

**애그리게이트 = 함께 다뤄야 하는 객체 묶음**

```
주문(Order) ← 애그리게이트 루트
  └── 주문상품(OrderItem) x 3개
  └── 배송지(Address)
  └── 결제정보(Payment)
```

이걸 **항상 같이** 저장하고 같이 조회해야 해. 주문상품만 따로 저장하면 안 됨.

**"애그리게이트 인스턴스"** = 이 묶음의 실제 데이터 하나.

이벤트 소싱에서 애그리게이트가 중요한 이유는, 이벤트 소싱은 한 번에 하나의 애그리게이트 인스턴스에 대한 이벤트를 질의할 수 있기 때문. (특정 주문 하나의 이벤트 이력은 조회 가능하지만, 여러 주문을 조건으로 검색하는 건 불가능)

## 11. 머터리얼라이즈

**"미리 계산해서 저장해두는 것"**

```
매번 조회할 때마다 계산:
SELECT SUM(price) FROM orders JOIN items ... (느림)

머터리얼라이즈:
미리 계산해서 → read_model 테이블에 저장
{ "user_id": 1, "total": 50000 }  (빠름)
```

CQRS에서 읽기 DB가 바로 이 머터리얼라이즈된 데이터야.

### 관계형 DB의 머터리얼라이즈 뷰와의 관계

원천 테이블(쓰기 DB)이 갱신되면 변경사항이 미리 작성된 뷰(읽기 DB)에 반영되어야 해. CQRS의 프로젝션이 이 역할을 함.

```
쓰기 DB (원천):
주문테이블: { user_id: 1, item: "신발", price: 50000 }
유저테이블: { user_id: 1, name: "유미" }

읽기 DB (머터리얼라이즈 뷰):
{ name: "유미", item: "신발", total: 50000 }
← 조인 미리 해서 저장해둠
```

원천이 바뀌면 → 뷰도 따라서 업데이트 = **프로젝션**

## 12. 프로젝션 - 동기식 vs 비동기식

### 동기식 vs 비동기식 기본 개념

**동기식** = 순서대로 기다림
```
나: "저장해줘"
시스템: 저장 중...
시스템: 읽기DB도 업데이트 중...
시스템: "다 됐어!"
나: 그제서야 다음 행동
```

**비동기식** = 던져놓고 안 기다림
```
나: "저장해줘"
시스템: "응 받았어!" (실제론 아직 처리중)
나: 바로 다음 행동
시스템: (백그라운드에서 읽기DB 업데이트 중...)
```

둘 다 사람이 직접 누르는 게 아니라 **자동**이야.

### 비동기식 프로젝션 작동 방식

```
커맨드 실행 모델 (쓰기DB)
    ↓ 변경사항 발행 (메시지 버스/우체통에 던짐)
변경사항 세트
    ↓ 구독 (읽기DB들이 우체통 확인)
프로젝션 엔진
    ↓ 생성
읽기 모델
```

**메시지 버스 = 우체통 같은 거:**
- 쓰기DB: "주문 생성됐어!" → 메시지 우체통에 투척 후 신경 안 씀
- 읽기DB들: 우체통 계속 확인하다가 메시지 발견하면 그때 업데이트

### 비동기식의 장단점

**장점:** 읽기 DB가 3개여도 쓰기는 빠름. 각자 알아서 나중에 처리하니까.

**단점 (도전과제):** 메시지 순서 꼬이면 읽기 DB 데이터가 엉망이 될 수 있음.

```
정상: 주문생성 → 결제완료 → 배송시작
꼬임: 배송시작 → 주문생성 → 결제완료  ← 읽기 모델 망가짐
```

취소된 주문인데 읽기DB엔 **"결제완료"** 로 보이는 상황이 발생.

### 비유

**동기식:** 카카오페이 송금하면 상대방 잔액 **즉시** 반영

**비동기식:** 은행 이체하면 **"처리 중"**이고 잠시 후 반영

### 결론

책에서는 **"동기식 먼저 구현하고, 필요하면 비동기식 추가해라"**고 권장.

| | 읽기DB 업데이트 시점 | 속도 | 안전성 |
|---|---|---|---|
| 동기식 | 즉시 | 느림 | 안전 |
| 비동기식 | 나중에 | 빠름 | 순서 꼬일 위험 |

## 13. CQRS 오해와 진실

### 오해하는 사람들 생각

> "CQRS니까 커맨드(쓰기)는 절대 데이터 반환하면 안 돼. 조회는 무조건 읽기 모델로만 해야 해!"

### 오해한 방식대로 하면 생기는 문제

쇼핑몰에서 주문 버튼을 눌렀다고 하자:

```
1. 주문 버튼 클릭 (커맨드)
2. 서버: "저장했어" (데이터 반환 없음)
3. 화면: 주문 결과 보여줘야 하는데...
4. 읽기 DB에 다시 조회 요청 (불필요한 왕복!)
5. 그제서야 "주문번호 1234, 배송지 부천시..." 보여줌
```

### 올바른 방식

```
1. 주문 버튼 클릭 (커맨드)
2. 서버: "저장했고, 결과는 이거야" (데이터 같이 반환)
3. 화면: 바로 주문 결과 보여줌. 끝.
```

커맨드가 결과 데이터를 돌려줘도 됨. 오히려 그게 더 자연스러워.

### 단 한 가지 제약

반환하는 데이터는 반드시 **쓰기 DB(커맨드 실행 모델)에서 비롯**되어야 해. 비동기식으로 업데이트되는 읽기 모델에서는 즉각적인 갱신을 기대할 수 없으니까.

## 14. CQRS가 이벤트 소싱에 적합한 이유

### 이벤트 소싱의 약점을 CQRS가 보완

이벤트 소싱은 조회가 불가능하다고 했잖아. CQRS가 그걸 해결해줘:

```
이벤트 소싱 (쓰기):
이벤트1: 회원가입
이벤트2: 포인트 +1000
이벤트3: 포인트 +500
이벤트4: 포인트 -200
    ↓
CQRS 프로젝션이 이벤트 읽어서 계산
    ↓
읽기 모델 (별도 DB):
| user_id | name | point |
|---------|------|-------|
|    1    | 유미 | 1300  |
    ↓
"포인트 1000 이상인 유저 찾아줘" → 바로 조회 가능!
```

**이벤트 소싱은 쓰기에 강하고 조회에 약한데**, CQRS가 **조회 전용 DB를 따로 만들어줘서** 약점을 보완.

### 전체 흐름 한눈에

```
[유저 행동]
    ↓
[커맨드] 저장 + 결과 바로 반환 ← 데이터 돌려줘도 됨!
    ↓
[이벤트 소싱] 이벤트로 쌓임 (조회 불가)
    ↓
[CQRS 프로젝션] 이벤트 읽어서 읽기 모델 생성
    ↓
[쿼리] 읽기 모델에서 빠르게 조회 가능
```
